The C Language Notes

Pointer

A.要声明不同类型的指针的原因？

  • 声明不同类型的原因： 对。不同类型的指针（如 int * 和 char *）存在的根本原因是为了告诉编译器，它所指向的对象有多大，这样才能正确地进行指针算术（如 ptr + 1）。
  • 指针的“值”没有区别： 对。在几乎所有现代计算机架构上，所有类型的指针（int *、char *、double *，甚至 void *）在内存中占据的空间大小都是一样的（例如 4 字节或 8 字节），并且它们存储的地址值本身也是相同的——它们都只是一个内存地址编号。

1. 指针的“值”（Value）是相同的

指针的“值”就是它存储的那个内存地址。对于同一个内存位置，无论是 int * 还是 char * 指向它，它们内部存储的地址编号都是一样的。

Pointer Value=Memory Address (e.g., 0x1000)

2. 指针的“类型”（Type）是不同的

指针的“类型”（如 int *、char *）是为了编译器服务的，它定义了如何解释指针所指向的内存，具体体现在两方面：

A. 内存步长（指针算术）

这是声明不同类型指针的主要原因。当您对指针进行递增或递减操作 (ptr++ 或 ptr + 1) 时，编译器需要知道要跳过多少字节。

指针类型 指向对象的大小 ptr + 1 移动的字节数
char * 1 字节 + 1 字节
int * 4 字节 (常见) + 4 字节
double * 8 字节 (常见) + 8 字节

结论： 声明 int * 是在告诉编译器：“这个地址指向一个 4 字节的块，所以下次移动时要跳过 4 个字节。”

B. 解引用类型（Dereferencing Type）

当您使用解引用运算符 * 时，编译器需要知道从该地址读取多少字节的数据，并将这些字节解释成什么类型的值。

指针类型 *ptr 行为 结果类型
char * 读取 1 个字节 解释为单个 char 值
int * 读取 4 个字节 解释为单个 int 值

3. void * (通用指针)

void * 进一步印证了您的观点。void * 是一种通用指针，它可以存储任何类型对象的地址。

  • void \* 的值： 依然是一个地址。
  • void \* 的类型： 它没有关联的类型信息（不知道指向的对象多大）。因此，您不能直接对 void * 进行指针算术 (ptr + 1) 或解引用 (*ptr)，因为它不知道该移动多少步或读取多少字节。

要使用 void *，您必须将其强制转换回具体的类型，比如 (int *)ptr，这样就重新赋予了它内存步长和解引用的规则。

B.p=&i和*p=i有什么区别

这是一个关于指针操作的两个基本、但意义完全不同的表达式。理解它们之间的区别是掌握C语言指针的关键。

假设我们已经声明了一个整数变量 i 和一个指向整数的指针变量 p：

C

现在我们来看这两个表达式：

1. 表达式：p = &i

操作 赋值 (Assignment)
&i 取地址运算符 (Address-of Operator)：获取变量 i 在内存中的地址。
p = ... 将右侧的值（地址）赋给左侧的指针变量 p。

含义和作用：

  • 目的：让指针 p 指向变量 i。
  • 结果：指针 p 现在存储了变量 i 的内存地址。
  • 谁变了？：指针 p 的值变了（它现在存储了一个地址）。变量 i 的值（10）没有改变。

总结：

p = &i 是建立联系。它使得指针 p 锁定了变量 i 的位置。

2. 表达式：*p = i

这个表达式的前提是：p 已经是一个有效的指针，并且已经指向了某个内存地址（例如，通过 p = &j 已经指向了另一个变量 j）。

操作 解引用赋值 (Dereference Assignment)
\*p 解引用运算符 (Dereference Operator)：访问指针 p 所指向的内存位置。
... = i 将变量 i 的值赋给左侧的内存位置（即 *p）。

含义和作用：

  • 目的：将变量 i 的值复制到指针 p 当前指向的内存位置。
  • 结果：p 所指向的变量的值被覆盖为 i 的值。
  • 谁变了？：指针 p 所指向的变量的值变了。指针 p 自己的值（它存储的地址）和变量 i 的值都没有改变。

示例场景：

假设变量 j 的地址被赋给了 p (p = &j)，并且 j 的初始值是 20。

内存示意图
执行前: i=10, j=20 且 p 指向 j。
执行后 (执行 \*p = i):
变量 i
变量 j
指针 p

总结：

\*p = i 是修改数据。它使得指针 p 通过其连接，去修改它所指向的变量的数据。

关键差异总结

表达式 操作对象 内存操作 作用
p = &i 指针本身 (p) 赋值一个地址 建立连接：让 p 知道 i 在哪里。
\*p = i p 指向的内存位置 (\*p) 赋值一个值 修改数据：把 i 的值写入 p 指向的地方。

地址调用

1. 内存中的“栈”（Stack）和“压栈”（Push）

“栈”是计算机内存中一个非常重要的区域，用于管理函数的调用和数据的存储。它遵循 后进先出 (LIFO) 的原则。

压栈的含义

“压栈”就是将数据放入栈顶端的过程。

在函数调用场景中，压栈有以下作用：

  • 保存返回地址： 将函数调用语句下一条语句的内存地址保存在栈中。这样，当函数执行完毕后，程序就知道要回到哪里继续执行。
  • 传递参数： 存储函数调用时传入的参数。
  • 分配局部变量： 为函数内部的局部变量预留存储空间。

简单例子：

假设主函数 main 在第 50 行调用了函数 funcA。

  1. 压栈 (Push): 在跳转到 funcA 之前，程序会将地址 51（即 main 函数中下一条语句的地址）压入栈中。
  2. 出栈 (Pop): funcA 执行完毕，程序从栈顶取出地址 51。
  3. 返回： 程序跳转到第 51 行，继续执行 main 函数的后续代码。

Data Structure

您的理解已经抓住了数据结构的核心。现在，让我们深入到内存管理层面，详细解释数据结构是如何影响数据在计算机内存中的“摆放”和“管理”的。

A.数据结构与内存管理

数据结构在内存管理上主要关注两种核心的内存分配方式和由此产生的逻辑组织。

一、 内存分配方式（物理存储）

计算机内存（RAM）可以看作是一个巨大的、连续的地址空间。数据结构决定了它们请求和使用这片空间的具体方式。

1. 连续存储 (Contiguous Allocation)

  • 代表结构： 数组 (Array)
  • 内存特性： 元素在内存中是紧挨着、连续存放的。
  • 管理方式：
    • 优点： 知道起始地址和元素大小，可以直接计算出任何一个元素的位置（地址 = 起始地址 + 索引 $\times$ 元素大小）。这使得随机访问（如获取 $A[5]$）的速度非常快（$O(1)$ 时间复杂度）。
    • 缺点： 必须在创建时就预留一大块连续的空间。如果内存碎片化严重，即使总空间足够，也可能找不到足够大的连续空间来创建数组。插入/删除操作非常慢，因为需要移动大量后续元素来保持连续性。

2. 离散存储 (Non-Contiguous Allocation)

  • 代表结构： 链表 (Linked List)
  • 内存特性： 元素（节点）可以分散在内存的任何地方。
  • 管理方式：
    • 如何连接： 每个节点不仅存储数据，还存储一个**指针（Pointer）**或**引用（Reference）**，指向下一个节点的内存地址。
    • 优点： 插入和删除操作非常高效（$O(1)$），只需要修改少数几个指针，不需要移动数据。内存利用率高，因为它不需要大块连续空间。
    • 缺点： 访问元素时不能直接计算地址，必须从头节点开始，沿着指针一个个查找，访问速度较慢（$O(N)$）。

二、 数据结构的逻辑组织（抽象管理）

除了底层的物理存储，数据结构还定义了数据的抽象访问规则，这才是栈、队列等概念的来源。

1. 栈 (Stack) - LIFO 规则

  • 逻辑管理： 强制只在一端（顶部，Top）进行数据的添加和移除。
  • 内存实现： 通常可以使用数组或链表实现。
    • 数组实现： 用一个索引变量来指示 Top 的位置。
    • 应用： 函数调用栈就是操作系统在内存中开辟的一个特定区域，用来管理函数执行流程。

2. 哈希表 (Hash Table) - 键值映射

  • 逻辑管理： 实现键（Key） 到 值（Value） 的快速映射。
  • 内存实现： 结合了数组和哈希函数。
    • 哈希函数： 将任意大小的 Key 转化为数组的一个有效索引（地址）。
    • 冲突处理： 当多个 Key 映射到同一个索引时（哈希冲突），通常使用链表将这些冲突的元素连接起来（称为拉链法）。
    • 目的： 追求近乎 $O(1)$ 的平均查找速度。

3. 树 (Tree) 和 图 (Graph) - 复杂关系管理

  • 逻辑管理： 组织具有层级或网络关系的数据。
  • 内存实现： 通常使用离散存储（节点和指针）或邻接矩阵/邻接表来实现。
    • 邻接矩阵（数组实现）： 用一个二维数组 $M[i][j]$ 的值来表示节点 $i$ 和节点 $j$ 之间是否有连接。适用于稠密图（边多）。
    • 邻接表（链表实现）： 使用一个数组，数组的每个元素是一个链表，链表中存储了所有与该节点直接相连的节点。适用于稀疏图（边少）。

数据结构不仅是关于“把数据怎么放”的，它更是一个蓝图，定义了：

  1. 物理层面： 它们是连续存放还是离散存放的。
  2. 逻辑层面： 它们之间的抽象关系（父子关系、前后关系、映射关系）。
  3. 操作层面： 保证在特定操作（查找、插入、删除）上达到最高的效率。

“抽象访问规则”正是将一种原始的存储方式（如数组或链表）转化为具有特定用途的抽象数据类型 (Abstract Data Type, ADT) 的关键。

让我详细解释一下这个概念，以及它是如何催生出栈、队列、字典这些我们熟悉的结构的。

B.抽象访问规则：从“物理”到“逻辑”

核心：物理实现 vs. 抽象模型

维度 物理存储 (Physical Implementation) 抽象访问规则 (Abstract Model)
关注点 如何实现：数据在内存中是连续的还是分散的。 如何使用：用户可以对数据进行哪些操作，以及这些操作遵循什么限制。
底层工具 数组、链表、指针。 抽象的指令（如 Push、Pop、Enqueue）。
概念层面 机器级别，具体的内存地址。 用户级别，概念化、理想化的操作模型。
举例 内存地址 0x1001 到 0x100A 连续存放了 10 个整数。 栈：只能从顶部存取数据；队列：从头部取，从尾部存。

抽象访问规则的本质

“抽象访问规则”就是一套对底层存储的限制和约束。它不是定义数据怎么放（那是物理存储的任务），而是定义数据怎么用。

这种抽象的目的是：将用户从复杂的内存管理细节中解放出来，专注于数据结构带来的逻辑功能。

C.数据结构与算法

1. 数据结构 (Data Structure)：组织和存放

数据结构本质上关注的是数据如何被组织和存储。您可以把它想象成一个仓库的设计图。

核心关注点：

  1. 数据的逻辑关系： 元素之间抽象的、概念上的关系（比如 LIFO、层级、网络连接）。
  2. 数据的物理存储： 元素在计算机内存中是连续存放（如数组），还是分散存放并通过指针连接（如链表）。
  3. 效率权衡： 如何摆放数据才能让特定的操作（如查找、插入）最快。

常见例子：

  • 栈 (Stack): 遵守后进先出 (LIFO) 规则。
  • 队列 (Queue): 遵守先进先出 (FIFO) 规则。
  • 哈希表 (Hash Table): 为了实现快速键值映射。
  • 树 (Tree): 用于组织层次化数据，便于高效搜索。

总结： 数据结构是名词，是存放数据的容器和组织形式。

2. 算法 (Algorithm)：解决问题的步骤

算法本质上关注的是如何操作数据来解决一个特定的问题。您可以把它想象成仓库里完成任务的操作手册。

核心关注点：

  1. 明确的步骤： 解决问题必须有一系列清晰、定义明确、有限的步骤。
  2. 输入和输出： 接收有效的输入，并在有限时间内产生期望的输出。
  3. 效率评估： 衡量完成任务所需的时间和资源（通常用时间复杂度和空间复杂度衡量）。

常见例子：

  • 排序算法: 如何将一组数据按升序或降序排列（如快速排序、归并排序）。
  • 搜索算法: 如何在一组数据中找到特定的元素（如二分查找）。
  • 图算法: 如何找到最短路径（如 Dijkstra 算法）。

总结： 算法是动词，是处理数据的步骤和方法。

Stack and Heap

A.抽象访问规则

栈和队列是最能体现“抽象访问规则”的例子，它们都基于线性结构（比如链表或数组）实现，但通过不同的规则定义了完全不同的行为和应用场景。

1. 栈 (Stack)：LIFO 规则的体现

  • 物理实现： 可以用一个链表来实现。
  • 抽象访问规则（LIFO 约束）：
    • 限制操作： 用户只能调用 Push（入栈）和 Pop（出栈）两个操作。
    • LIFO 约束： Push 必须作用于链表的头部（或数组的末端）。Pop 也必须作用于同一位置。
  • 总结： 栈将底层的链表或数组抽象化成了一个“只能一端进出”的容器。用户不需要关心它是数组还是链表，只需要知道它是一个 LIFO 结构。

2. 队列 (Queue)：FIFO 规则的体现

  • 物理实现： 也可以用一个链表来实现。
  • 抽象访问规则（FIFO 约束）：
    • 限制操作： 用户只能调用 Enqueue（入队）和 Dequeue（出队）。
    • FIFO 约束： Enqueue 必须在队尾添加元素。Dequeue 必须从队头移除元素。
  • 总结： 队列将底层的链表或数组抽象化成了一个“两端进出”的管道。这种抽象模型完美地模拟了现实世界中的排队行为。

其他结构的抽象和用途

这种“定义抽象访问规则”的思维方式适用于所有高级数据结构：

数据结构 底层物理实现（示例） 抽象访问规则/约束 提供的核心功能
哈希表 (Hash Table) 数组 + 链表 必须通过**键（Key）**访问数据；Key 必须经过**哈希函数**转换成索引。 极速的 $O(1)$ 查找和插入。
树 (Tree) 节点 + 指针（链式结构） 每个节点最多有一个父节点；必须遵循特定的遍历顺序（如中序、前序、后序）。 高效的层次化组织和搜索（如二叉搜索树）。
图 (Graph) 节点 + 指针/矩阵 允许节点之间建立任意连接（边）；必须通过特定的搜索算法（DFS/BFS）来访问。 建模复杂的网络关系（如社交关系、地图路径）。

正是这些“抽象访问规则”（即对底层操作的约束），使得数据结构从单纯的内存分配技巧（数组、链表）提升为强大的抽象数据类型（栈、队列、字典）。

它们规定了“数据的玩法”，而这种玩法是程序员解决特定问题的强大工具。您在使用 Stack 时，不需要知道它内部是用什么实现的，您只需要知道它遵守 LIFO 规则，这便是抽象的力量。

B.具体实体

它们具体的实体是什么，取决于您在编写代码时选择了哪种物理实现：

实体形式一：基于 数组 的实现

如果您用数组（Array）来构建栈或队列：

  • 具体实体： 栈/队列就是内存中一块连续的、编号整齐的空间。
  • 具体管理：
    • 程序会设置一个或两个整数变量（比如 top 或 head 和 tail），这些变量作为指针，告诉程序数据应该放在哪里、从哪里取出。
    • 数据本身（比如数字、字符、对象）就一个个紧密地排列在这块内存空间里。

实体形式二：基于 链表 的实现

如果您用链表（Linked List）来构建栈或队列：

  • 具体实体： 栈/队列是由多个**散落在内存各处的小盒子（节点）**组成的。
  • 具体管理：
    • 每个小盒子包含两部分：您的数据和下一个小盒子的地址（即指针）。
    • 程序会设置一个或两个指针变量（比如 头指针 和 尾指针），它们存储着第一个和/或最后一个小盒子的内存地址。
    • 这些小盒子不需要在内存中连续，它们通过互相指引的地址连接起来。

总结：抽象和具体的结合

所以，栈和队列是具体的实体，它们是：

  1. 物理层面： 一串连续的字节（如果是数组实现），或者一系列由指针连接的节点（如果是链表实现），这些都是存在于**计算机内存（RAM）**中的真实数据。
  2. 抽象层面： 在这些物理实体之上，编程语言的代码又强制性地赋予了它们 LIFO (栈) 或 FIFO (队列) 的访问规则。

它们既是“抽象概念”，又是“具体实体”，这种双重身份正是数据结构的精髓所在。

  • 抽象概念（LIFO/FIFO）：告诉您怎么使用它们。
  • 具体实体（内存中的数组/链表）：告诉您它们是怎么造出来的。

对于一个程序员来说，当他说“我创建了一个栈”时，他的意思是：他在内存中分配了空间，并且写下了保证 LIFO 规则的代码。

C.基于数组和链表实现的区别

栈（Stack）和队列（Queue）都可以基于数组和链表这两种底层物理存储方式来实现。它们的主要区别在于**性能（时间复杂度）**和**空间效率**上的权衡。

栈 (Stack) 的实现对比：数组 vs. 链表

栈的操作集中在顶部（Top），对应数组的末端或链表的头部。

特性 基于数组的栈 (Array-based Stack) 基于链表的栈 (Linked List-based Stack)
物理存储 连续的内存块。 离散的内存块，通过指针连接。
Push/Pop 效率 $O(1)$ $O(1)$
空间效率 高：没有额外的指针开销。 低：每个元素都需要额外的空间存储一个指针。
内存开销 存在空间浪费（预先分配过多）或空间不足（需要动态扩容）的风险。 没有浪费：内存按需分配，但分配和释放内存有时间开销。
动态性 当数组满时，必须进行扩容操作（通常是复制到一个更大的数组），扩容成本高昂（$O(N)$）。 天然动态：只要内存允许，可以无限增长，不需要扩容操作。

总结：

  • 数组栈： 适用于元素数量已知或变化不大的场景。它的读写速度快，但需要处理扩容问题。
  • 链表栈： 适用于元素数量变化巨大的场景。它灵活且能按需使用内存，但牺牲了额外的空间来存储指针。

队列 (Queue) 的实现对比：数组 vs. 链表

队列的操作涉及**队头（Dequeue）和队尾（Enqueue）**两端。

特性 基于数组的队列 (Array-based Queue) 基于链表的队列 (Linked List-based Queue)
物理存储 连续的内存块。 离散的内存块，通过指针连接。
Enqueue (入队) $O(1)$ (在数组末尾) $O(1)$ (如果维护了尾指针)
Dequeue (出队) $O(1)$ (如果使用循环数组实现)；否则为 $O(N)$ (如果每次出队后都要移动所有元素填补空位)。 $O(1)$ (在链表头部)
空间效率 同样存在扩容风险和空间浪费问题。循环数组可以解决浪费，但逻辑更复杂。 低：每个元素都需要额外的空间存储一个指针。
实现难度 高：实现高效的**循环队列（Circular Queue）**以避免 $O(N)$ 的移动开销和内存浪费，逻辑相对复杂。 低：实现简单，只需要维护头指针和尾指针即可。

总结：

  • 数组队列： 只有通过循环数组的精巧设计，才能保证 Dequeue 操作也是 $O(1)$。这种实现最快，但代码难度较高。
  • 链表队列： 实现最简单，性能最稳定，所有核心操作都是 $O(1)$，是实际开发中最常用的队列实现方式。

最终结论：实际选择倾向

数据结构 最佳选择及原因
栈 (Stack) 通常选择基于数组的实现。因为栈的操作都在一端，数组的连续存储提供了更高的空间效率和缓存优势。
队列 (Queue) 通常选择基于链表的实现。因为它自然地支持在两端（头和尾）进行 $O(1)$ 操作，避免了数组实现中复杂的循环队列逻辑和内存移动问题。

一.Fgets

fgets 是 C 语言标准库 <stdio.h> 中的一个函数，用于从指定的流（stream）中读取一个字符串。它非常常用，特别是在需要读取包含空格的一整行输入时。

🔍 fgets 函数解释

1. 函数原型 (Function Prototype)

C

2. 参数说明

参数 类型 描述
str char * 目标缓冲区指针。这是一个指向字符数组的指针，fgets 会将读取到的数据存储到这个数组中。
size int 最大读取字节数。这是目标缓冲区的大小（包括末尾的空字符 \0）。fgets 最多读取 $size - 1$ 个字符。
stream FILE * 输入流指针。指定从哪里读取数据。最常用的流包括： - stdin：标准输入流（通常是键盘）。 - 文件指针：指向已打开文件的指针。

3. 工作原理和特点

fgets 的读取规则非常严格，这也是它比 gets（已弃用，不安全）和 scanf("%s") 更安全、更适合读取整行的原因：

  • 读取停止条件：fgets 会在以下任一条件满足时停止读取：
    1. 读取到换行符（\n）。
    2. 读取了 $size - 1$ 个字符。
    3. 到达文件末尾（EOF）。
  • 包含换行符：如果读取到了换行符（\n），fgets 会将这个换行符也存储到目标字符串 str 中。
  • 自动添加空字符：无论读取因何种原因停止，fgets 都会在字符串的末尾自动添加一个空字符（\0），确保它是一个合法的 C 字符串。
  • 防止缓冲区溢出 (Buffer Overflow)：通过 $size$ 参数，fgets 限制了它能写入目标缓冲区的最大字节数，这使其成为一个安全的输入函数。

4. 返回值

  • 成功：返回 str（即目标缓冲区的指针）。
  • 失败/文件末尾：如果在读取任何字符之前遇到文件末尾（EOF）或发生错误，返回 NULL。

二.fgets 与 scanf（%s）的区别

特点 fgets scanf(“%s”)
是否读取空格 能。 它会读取整行，直到遇到换行符 \n，因此行内的空格会被完整保留。 不能（默认情况下）。 它默认以空白字符（空格、制表符 \t、换行符 \n）作为分隔符来读取数据。
是否安全 安全。 它通过 size 参数限制读取的最大长度，有效防止缓冲区溢出。 不安全（老版本和无限制时）。 如果不指定最大宽度（例如只用 scanf("%s", str);），可能导致缓冲区溢出。安全的用法是 scanf("%99s", str);。
是否包含换行符 包含。 如果读取到了换行符 \n，它会将其存储在字符串中（通常需要手动移除）。 不包含。 它在读取到换行符或空格时停止，并将该空白字符留在输入缓冲区中。
用途 读取一整行文本，包括行内的所有空格。 读取单个词汇/标记或特定格式的数据（如整数、浮点数）。

🔑 总结核心区别

  1. fgets (面向行)：它将输入视为一个整体的行来处理。它看到空格会继续读，只有遇到行末的换行符 \n 或达到缓冲区大小限制时才会停止。
  2. scanf("%s") (面向标记)：它将输入视为由空格分隔的一系列标记（token）。它一旦遇到空格、制表符或换行符，就认为当前标记读取结束并停止。

三.函数与返回值

这是一个很好的问题，但你的问法可能有点混淆了两个概念：函数本身和函数的返回值。

让我们来明确一下：

1. fgets 本身是一个函数

fgets 是一个执行特定操作（从流中读取数据）的代码块。在 C 语言中，任何执行任务的代码块，比如 main、printf、scanf 都是函数。

  • 结论： fgets 是一个函数。

2. fgets 的返回值是一个指针

fgets 函数的返回值类型是 char *（字符指针）。

  • 结论： fgets 的返回值是一个指针。

澄清误解

你可能想问的是：

“为什么我们说 fgets 是指针函数（返回指针的函数），而不是说它返回一个普通的值（比如 int 或 void）？”

正如前面所解释的，fgets 必须返回一个指针（char *）主要有两大原因：

🎯 目的 A: 状态报告 (返回 NULL)

fgets 的设计目的是告诉调用者操作是否成功或是否到达了文件末尾。

  • 如果成功读取一行，返回一个非空地址（即缓冲区的地址）。
  • 如果遇到 EOF 或发生错误，它返回一个特殊值 NULL (空指针)。

使用指针作为返回值，可以同时传递两种信息：成功时的数据位置，和失败时的终止信号。如果它只返回一个 int（比如错误码），那么我们就需要另一个方法来获取刚刚读取的字符串的地址，代码会更复杂。

🎯 目的 B: 方便链式操作 (返回 buffer 地址)

当 fgets 成功时，它返回其第一个参数 str（即缓冲区 buffer 的地址）。这使得该函数的调用可以在表达式中直接使用。

总结

  • fgets 是函数：因为它执行一个操作。
  • fgets 返回指针：因为它需要一个灵活的值来：
    1. 在成功时，返回刚刚填充的缓冲区的地址。
    2. 在失败或结束时，返回终止信号 NULL。

scanf 和 printf 都是有返回值的，它们的返回值都是 int 类型，用于指示函数执行的状态和成功处理的项数。

1. printf 的返回值

printf 返回的是成功写入输出流的字符总数（不包括末尾自动添加的空字符 \0）。

返回值 含义 示例
正整数 成功写入的字符数。 int count = printf("Hello\n"); $\rightarrow$ count 的值是 $6$（’H’, ‘e’, ‘l’, ‘l’, ‘o’, \n）。
负整数 发生输出错误。 写入磁盘文件失败等情况下返回 EOF（通常是 $-1$）。

2. scanf 的返回值

scanf 返回的是成功匹配并赋值的输入项的个数。

返回值 含义 示例
$N$ (正整数) 成功匹配并赋值了 $N$ 个输入项。 scanf("%d %f", &i, &f); 如果用户输入 10 3.14，返回 $2$。
$0$ 成功匹配了格式，但没有成功赋值任何项。这通常发生在输入的数据类型与格式字符串不匹配时。 scanf("%d", &i); 如果用户输入了 hello，返回 $0$。
EOF (负整数) 在读取任何数据之前，就达到了文件结束（EOF）或发生了读取错误。 用户按下 Ctrl+D 或 Ctrl+Z 时返回 EOF（通常是 $-1$）。

四.arr,arr[0],arr[0] [0]

在一个 C/C++ 程序中，如果声明了一个二维数组 int arr[100][100];，那么单独使用数组名 arr 而不带任何下标时，它代表的含义是：

🔑 arr 的含义：指向数组的指针

arr 表达式在大多数情况下会退化（decay）为一个指向其第一个元素的指针。

由于 arr 是一个 100 行 100 列的二维数组，它的第一个元素是第一行（即 arr[0]）。而第一行本身是一个包含 100 个整数的一维数组。

因此，arr 退化后的类型是：

$$\text{类型: } \mathbf{\text{int} (*)[100]}$$

这意味着：arr 是一个指向包含 100 个整数的数组的指针。

示例和对比

为了更好地理解，我们来对比 arr、arr[i] 和 &arr[0][0] 的含义：

表达式 类型 (Type) 含义 (Meaning)
arr int (*)[100] 整个二维数组的首地址，或 指向第一行（一个含有 100 个整数的数组）的指针。
arr[0] int * 第一行的首地址，或 指向第一行第一个元素（即 arr[0][0]）的指针。这是由 arr[0]（一个一维数组）退化得来的。
&arr int (*)[100][100] 整个二维数组的指针（通常与 arr 的值相同，但类型不同）。
&arr[0][0] int * 数组中第一个整数的地址。

下面是梳理思路和入手的步骤：

🐍 第一步：核心数据结构 (Snake)

贪吃蛇游戏最关键的部分是表示蛇的身体。由于蛇会不断变长，并且需要跟踪身体的每个节的位置，链表是理想的选择。

2. 链表管理

你需要两个指针来管理整个蛇的身体：

  • head (蛇头)：指向链表的第一个节点，代表蛇运动的最前端。
  • tail (蛇尾)：指向链表的最后一个节点，在蛇移动或吃食物时需要修改。

🗺️ 第二步：游戏环境和状态

1. 游戏地图 (Game Map)

使用一个二维数组表示游戏区域，例如 $20 \times 40$ 的格子：

  • 存储内容： 数组中存储字符来表示格子中的内容：
    • ' '：空白区域
    • '#'：墙壁/边界
    • '@'：食物 (Food)
    • 'O'：蛇的身体/头部

2. 游戏状态和变量

需要跟踪和管理的关键变量：

  • int score：当前得分。
  • int food_x, food_y：食物的坐标。
  • int direction：当前的移动方向（例如，用数字 0, 1, 2, 3 分别代表上、下、左、右）。
  • int game_over：布尔标志，用于判断游戏是否结束。

⚙️ 第三步：主要函数和算法

你的程序将由以下几个关键函数组成：

1. 🚀 初始化 (initializeGame)

  • 设置 game_map 的边界。
  • 将 head 和 tail 初始化为包含一个或几个节点（蛇的初始长度）。
  • 随机生成初始食物位置 (spawnFood)。

2. 🔄 游戏循环 (mainGameLoop)

这是游戏的心脏，它会不断重复以下步骤，直到 game_over 为真：

  1. 处理输入 (handleInput)：检查用户按下的方向键，更新 direction 变量。
  2. 更新游戏状态 (updateGame)：根据 direction 计算新的蛇头位置，执行移动逻辑。
  3. 渲染屏幕 (drawGame)：清空屏幕并打印 game_map 的当前状态。

3. 🐍 移动逻辑 (Movement)

这是最复杂的步骤，也是头插尾删的应用：

  1. 头插 (创建新的蛇头)：根据当前方向 (direction)，在新的坐标 $(x’, y’)$ 上创建一个新节点。

    C

  2. 碰撞检测：检查新的蛇头位置 $(x’, y’)$ 是否撞到墙壁、撞到自己 (game_over = 1) 或撞到食物。

4. 🍎 吃食物逻辑 (Eating Food)

如果新蛇头撞到了食物 $(food_x, food_y)$：

  • 蛇变长： 跳过步骤 5 (不删除蛇尾)，蛇身自然变长一节。
  • 得分： 增加 score。
  • 重新生成食物： 随机生成新的食物位置。

5. ✂️ 移动和变短 (尾删)

如果新蛇头没有撞到食物，蛇身长度保持不变，需要删除尾部节点：

  • 找到倒数第二个节点 (pre_tail)。
  • 释放掉 tail 节点占用的内存 (free(tail))。
  • 更新 pre_tail->next = NULL。
  • 将 tail 更新为 pre_tail。

⌨️ 第四步：输入和屏幕刷新

在标准的 C/C++ 命令行环境下，你需要使用特定的方法来处理输入和屏幕刷新：

  • 输入： 使用 conio.h 中的 kbhit() 和 getch() (在 Windows 上) 或 <termios.h> (在 Linux 上) 来实现非阻塞式输入，这样程序就不用等待用户按键。
  • 清屏： 使用 system("cls"); (Windows) 或 system("clear"); (Linux/macOS) 来清空控制台，实现画面刷新。
